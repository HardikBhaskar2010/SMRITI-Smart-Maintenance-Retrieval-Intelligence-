## 2. Functional Requirements

### FR-01: Document Ingestion Pipeline

**Priority:** P0 (must have for demo)

- System must accept PDF, DOCX, and scanned image files via drag-and-drop upload
- Ingestion pipeline must extract raw text using PyMuPDF (PDF) and python-docx (DOCX)
- OCR fallback via pytesseract for scanned pages
- Claude API must identify and tag all equipment identifiers (e.g. UPS-01, T-101, P-207) within the extracted text
- Each identified equipment tag must create or update a dedicated ChromaDB collection
- System must display real-time ingestion progress: files processed, tags found, items added per thread
- Ingestion of a 50-page PDF must complete in under 45 seconds

**Acceptance criteria:**
- Upload a Schneider UPS manual → at minimum 3 distinct equipment threads created
- Each thread item has: text chunk, source document, page number, timestamp, confidence score
- Duplicate content across documents is detected and not re-embedded

### FR-02: Asset-Thread RAG Query Engine

**Priority:** P0

- Each equipment tag must have a dedicated retrieval context (its "thread") stored in ChromaDB
- On query, system must identify which equipment tag(s) are referenced in the query
- System must retrieve all relevant items from the identified asset thread(s)
- Claude API must receive the full thread context and produce a response with source citations
- Citations must include: source document name, page/section reference, date of document
- Multi-asset queries must be supported (e.g. "compare UPS-01 and UPS-02")
- Query response time must be under 8 seconds for a thread with up to 200 items

**Acceptance criteria:**
- Query "UPS-01 mein last month kya issues aaye?" returns incidents sourced from UPS-01's thread
- Each response item includes a clickable source citation
- An unknown equipment tag returns a graceful "no thread found" response

### FR-03: Knowledge Debt Score

**Priority:** P0

- System must calculate a Knowledge Debt Score (0–100) for every equipment tag in the Asset-Thread Store
- Score must be computed from four weighted dimensions:
  - Document coverage: 30%
  - Expert distribution: 35%
  - Recency of knowledge: 20%
  - Operational criticality (manually set or inferred): 15%
- Score must update in real time when new knowledge is added (e.g. after Guru Mode session)
- Severity thresholds: OK (0–40), WARNING (41–70), CRITICAL (71–100)
- A dashboard API endpoint must return all asset scores, sortable by severity

**Acceptance criteria:**
- T-101 with 1 expert and sparse documentation scores CRITICAL (>70)
- UPS-01 with 3 experts and full documentation scores OK (<40)
- Score changes within 2 seconds of new knowledge being embedded

### FR-04: Guru Mode — Expert Interview Engine

**Priority:** P0

- Guru Mode must be initiatable for any asset thread marked WARNING or CRITICAL
- System must generate a set of targeted interview questions derived from knowledge gaps in the asset's thread
- Questions must be generated by Claude API using the asset's existing thread as context
- Interview must proceed as a structured back-and-forth: system asks, expert answers, system asks follow-up
- Expert responses must be embedded and added to the asset's thread in real time
- Knowledge Debt Score must update visibly after each response is embedded
- Session transcript must be saved and accessible as a permanent audit record

**Acceptance criteria:**
- Start Guru Mode on T-101 → system generates at least 5 relevant questions about T-101 specifically
- Answer 3 questions → T-101's Knowledge Debt Score decreases by at least 10 points
- Session transcript is stored and retrievable after the session ends

### FR-05: Voice Query Interface

**Priority:** P1

- System must use browser-native Web Speech API (no third-party dependency)
- Must support speech recognition in: English, Hindi, Hinglish (mixed Hindi-English)
- Voice input must be triggered by a single button press
- Recognized text must appear in the query field before submission (editable)
- Recognition language must auto-detect or default to `hi-IN` with English fallback
- If voice is unavailable (permissions denied), system must fall back to text input gracefully

**Acceptance criteria:**
- Speak "UPS-01 mein last month kya issues aaye?" → query field populates with transcription → result returned
- Works on Chrome desktop and Chrome mobile without installing additional software

### FR-06: 3D Knowledge Graph Visualization

**Priority:** P1

- Graph must render all equipment tags as 3D nodes using React Three Fiber
- Node size must be proportional to the number of items in that asset's thread
- Node color must encode Knowledge Debt Score: green (OK), amber (WARNING), red (CRITICAL)
- Nodes must be interactive: hover shows tooltip with asset name + score, click opens thread detail panel
- Graph must animate smoothly when new nodes are added (ingestion) or when scores change (Guru Mode)
- Must run at ≥ 30fps on a mid-range laptop GPU

**Acceptance criteria:**
- 10+ nodes render without frame rate drop below 30fps
- Clicking T-101 node opens its thread detail panel
- After Guru Mode session, T-101 node color transitions from red to amber in real time

### FR-07: Progressive Web App (PWA)

**Priority:** P2

- App must be installable as a PWA on desktop and mobile
- Core UI (dashboard, query interface) must function offline using cached data
- Service worker must pre-cache the last-known asset thread data
- New queries while offline must be queued and executed when connectivity returns

## 3. Non-Functional Requirements

### NFR-01: Performance

| Operation | Target |
|---|---|
| App initial load (first paint) | < 2 seconds |
| Document ingestion (50-page PDF) | < 45 seconds |
| Voice query end-to-end response | < 8 seconds |
| Knowledge Debt Score recalculation | < 2 seconds |
| 3D graph render (20 nodes) | ≥ 30 fps |

### NFR-02: Reliability

- System must handle API timeout from Claude/OpenRouter gracefully with a retry (max 2 retries)
- ChromaDB must persist to disk so data survives server restarts
- All ingested data must be recoverable after a crash without re-uploading documents

### NFR-03: Compatibility

- Frontend must render correctly on Chrome 120+, Firefox 120+, Safari 17+
- Responsive layout required: 1920px desktop, 1440px laptop, 768px tablet, 390px mobile
- Voice input (FR-05) is Chrome-only — clearly communicated to user on other browsers

### NFR-04: Security (Prototype Scope)

- No user authentication required for hackathon prototype
- API keys (OpenRouter) must be stored in environment variables, never committed to source
- No PII ingested or stored — industrial documents only

    "document_coverage": 0.35,

### Knowledge Debt Calculation

```python
def calculate_debt_score(asset):
    doc_score     = (1 - asset.document_coverage) * 30   # max 30
    expert_score  = max(0, (1 - (asset.expert_count / 3))) * 35  # max 35
    recency_score = calculate_recency_penalty(asset.last_updated) * 20  # max 20
    crit_score    = asset.operational_criticality * 15  # max 15
    return min(100, doc_score + expert_score + recency_score + crit_score)
```

## 5. API Specifications

### Backend Endpoints (FastAPI)

| Method | Endpoint | Description |
|---|---|---|
| POST | `/api/ingest` | Upload document, trigger thread build |
| GET | `/api/assets` | List all asset threads with debt scores |
| GET | `/api/assets/{id}` | Get full thread for a single asset |
| POST | `/api/query` | Submit text query, returns RAG response |
| POST | `/api/guru/start` | Start Guru Mode session for an asset |
| POST | `/api/guru/respond` | Submit expert answer, embed and update score |
| GET | `/api/guru/session/{id}` | Get full session transcript |
| GET | `/api/graph` | Get node + edge data for 3D graph |

### LLM API Usage (OpenRouter — Free Tier)

SMRITI routes to **two different free models** depending on the workload:

| Role | Model (OpenRouter ID) | Max Output Tokens |
|---|---|---|
| Document tag extraction | `cohere/north-mini-code:free` | 2,048 |
| Asset-Thread RAG query | `meta-llama/llama-3.3-70b-instruct:free` | 4,096 |
| Guru Mode interview | `meta-llama/llama-3.3-70b-instruct:free` | 1,024 |
| Query fallback (if Llama 3.3 rate-limited) | `poolside/laguna-xs2:free` | 4,096 |

**Model selection rationale:**
- `cohere/north-mini-code:free` — 19.8B code-trained model; 707ms median latency, 89 t/s. Code training = strongest JSON-only instruction adherence for extraction pipeline.
- `meta-llama/llama-3.3-70b-instruct:free` — Proven #1 open-source multilingual model; native Hindi/Hinglish support required for voice query demo. 131k context handles full asset threads.
- All models are **$0/token** on OpenRouter free tier.
